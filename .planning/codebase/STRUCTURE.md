## Directory Purposes

**Repo root:**
- Purpose: houses the single-file web app (`index.html`) and top-level deploy config.
- Contains: `index.html`, `README.md`, `wrangler.toml`.
- Key files: `index.html` — this IS the application; there is no `src/` directory.

**`worker/`:**
- Purpose: Cloudflare Worker that proxies AI chat/vision requests so no API key ships to the browser.
- Contains: `anthropic-proxy.js` (Worker logic), `wrangler.toml` (Worker-scoped deploy config used when deploying with root directory = `worker`), `README.md` (setup/deploy docs).
- Key files: `worker/anthropic-proxy.js`.

**`android/`, `ios/`:**
- Purpose: native app shells generated by Capacitor to package `index.html` as a mobile app (WebView wrapper). Not hand-written application logic.
- Contains: platform project files, generated copies of the web bundle, plugin glue code.
- Key files: `ios/App/App/public/` and `android/app/src/main/assets/` (or equivalent) hold the deployed copy of `index.html`; these are build artifacts synced via Capacitor CLI (`npx cap sync`), not edited directly.
- Generated: Yes (regenerate via Capacitor tooling rather than hand-editing).

**`node_modules/`:**
- Purpose: npm dependencies for Capacitor tooling (mobile build only) — the web app itself has zero runtime npm dependencies (all JS is inline, all external libs loaded via CDN `<script>`/`<link>` tags in `index.html:1-9`, e.g. Tabler Icons, Google Fonts, Firebase compat SDK).
- Generated: Yes. Committed: No (should be gitignored).

## Key File Locations

**Entry Points:**
- `index.html`: the entire app — HTML shell, `<style>` theme block, `<script>` containing all state/logic/render, and the bootstrap IIFE at the bottom (`index.html:5141-5158`) that kicks off session restore and first `render()`.
- `worker/anthropic-proxy.js`: Cloudflare Worker HTTP entry point (separate runtime, deployed independently).

**Configuration:**
- `wrangler.toml` (root) and `worker/wrangler.toml`: Cloudflare Worker/Pages deploy configuration.
- `_fbCfg` object inside `index.html:905`: Firebase project config (apiKey, databaseURL, etc.) — inline, not in a separate env file.
- `AI_PROXY` variable (`index.html:293`): Worker URL the browser calls for AI requests; blank by default (falls back to user-supplied Anthropic key or no AI).

**Core Logic (all within `index.html`, no separate files):**
- State: `S` object, `index.html:1625`.
- Render loop: `render()`, `index.html:1809`.
- DOM builder: `el()`, `index.html:1213`.
- Persistence: `DB`, `index.html:897`.
- Firebase init: `index.html:905-907`.
- Views: functions named `v*` throughout `index.html` (see ARCHITECTURE.md for full list with line numbers).
- i18n: `I18N` object and `t()`/`L()` helpers, `index.html:1656` onward.

**Testing:**
- Not present. No test framework, test files, or test config exist in the repo.

## Naming Conventions

**Files:**
- Single lowercase-hyphenated or plain names: `index.html`, `anthropic-proxy.js`, `wrangler.toml`. No per-feature file split — everything lives in `index.html`.

**Functions:**
- View/screen functions: `v` prefix + PascalCase suffix, e.g. `vDash`, `vNutr`, `vSport`, `vGang`, `vAI`, `vAdmin` — one per screen/section.
- Helper/utility functions: lowerCamelCase, e.g. `getLvl`, `getXP`, `pushClanEvent`, `setupGangListener`, `bootAccount`.
- Internal/private-ish helpers prefixed with underscore: `_userKey`, `_hashPw`, `_safeKey`, `_enterApp`, `_checkPassedYou`.

**Variables:**
- Global singletons: short uppercase or capitalized identifiers for cross-cutting concepts — `S` (state), `DB` (storage), `RTDB` (Firebase handle), `UID`, `PCODE`, `TODAY`, `CR`, `NAV`, `GOALS`, `ACT`, `OBS`.
- Draft/input state fields on `S`: short abbreviated camelCase, e.g. `wIn`, `slH`, `slQ`, `slD`, `mIn`, `wtIn`, `bfIn`, `fqty`, `chatIn` — used for uncommitted form input tied to a specific log type.
- Persisted domain collections on `S`: full plural words, e.g. `food`, `workouts`, `waterLog`, `sleepLog`, `measurements`, `photos`, `doneChals`, `chat`, `suppLog`.

**Types:**
- No TypeScript / no formal type system — plain JS objects and arrays throughout; shape is implied by usage, not declared.

## Where to Add New Code

**New Feature (new screen/section):**
- Add a new `v<Name>()` function in `index.html` following the existing pattern (build DOM tree via `el()`, read from `S`, wire handlers that mutate `S` then call `render()`).
- Register it in the nav array (`NAV`/`_NAV_N`/`_NAV_B` near `vMain`, `index.html:2140` region) so it appears in the sidebar/mobile nav, and add a branch in whatever view dispatches to it (`vMain`'s internal `S.nav` switch).
- If the feature needs new persisted data, add a new key to `S` (persisted collection naming convention above) and load/save it via `DB.load`/`DB.save`.
- Per user's global CLAUDE.md instruction: any new user-facing feature must also be added to the in-app guided tour (`tourSteps`) — search `index.html` for `tourSteps` before considering a feature complete.

**New Component/Module:**
- There is no component abstraction beyond `v*` functions and small DOM-builder helper functions (like `mkGauge`, `ic`). Add new reusable UI fragments as a plain function `function mkX(...){ return el(...) }` near related helpers, and call it from the relevant view function(s).

**Utilities:**
- Shared helpers (calculations, formatting, i18n) live as top-level functions/objects in `index.html` (e.g. `Calc` object, `t()`/`L()`, `getToday()`). Add new pure helpers alongside similar existing ones rather than creating new files.

**Cloud data:**
- New Firebase-synced data should follow existing path conventions under `RTDB`: `users/<key>/...` for per-account private data, `sh/gang/<code>/...` for shared clan data, `accounts/<key>` for auth records, `friends/`/`friendReqs/`/`notifs/` for social graph data.

## Special Directories

**`worker/`:**
- Purpose: independently deployable Cloudflare Worker (separate JS runtime/deploy target from the main web app).
- Generated: No — hand-written.
- Committed: Yes.

**`android/`, `ios/`:**
- Purpose: Capacitor-generated native app projects wrapping the web app for app-store distribution.
- Generated: Yes (via `npx cap add android` / `npx cap sync`), including the copied web assets inside `android/app/src/main` and `ios/App/App/public`.
- Committed: Appears committed (present in working tree) — treat as build output; prefer regenerating via Capacitor CLI over hand-editing generated subfolders, only edit platform-specific native config files (manifests, Info.plist, etc.) directly when needed.

**`node_modules/`:**
- Purpose: Capacitor CLI/tooling dependencies for building the mobile shells.
- Generated: Yes.
- Committed: No (should remain gitignored; not part of the deployed web app).
